# 1.5 权限与安全文档

## 1. 安全目标

PetroChat-Agent 当前是工程演示与能力验证项目，但设计上需要体现生产化安全意识：

1. 不泄露 API Key。
2. 不允许 LLM 直接执行危险 SQL。
3. 不写业务数据库。
4. 不把图表 base64 塞进 LLM 上下文。
5. 不把用户会话和长期记忆无边界混用。
6. 前端清晰展示错误，不吞掉失败原因。

## 2. 密钥与配置安全

| 配置 | 用途 | 安全策略 |
| --- | --- | --- |
| `DEEPSEEK_API_KEY` | Chat / Reasoner 模型 | `.env`，不提交 |
| `DASHSCOPE_API_KEY` | Embedding | `.env`，不提交 |
| `MYSQL_PASSWORD` | 只读数据库连接 | `.env`，不提交 |
| `LANGSMITH_API_KEY` | Trace | `.env`，不提交 |

`.gitignore` 已忽略：

- `.env`
- `.env.local`
- `.env.*.local`
- `data/runtime/`
- `data/ducuments/`
- `frontend/node_modules/`
- `frontend/dist/`

`/config` 只返回 key 是否配置，不返回明文。

## 2.1 当前阶段确认口径

| 项目 | 当前决策 | 后续生产化建议 |
| --- | --- | --- |
| 用户密码 | 暂按 `user.password` 明文比对 | 改为 bcrypt/argon2 等哈希存储 |
| 登录态 | 前端 localStorage 保存本地演示 token | 改为后端签发 JWT 或 HttpOnly Cookie Session |
| 工程师数据范围 | 当前先允许白名单表全量只读查询 | 后续按部门、区域、字段或数据权限规则收敛 |
| 管理员删除异常数据 | 暂不需要二次确认或审批流 | 高风险操作增加二次确认、审批或留痕复核 |
| LangSmith / 普通日志 | 生产默认不保留完整真实问题、完整 SQL、完整检索片段 | 只保留元数据、摘要、ID、指标和错误信息；真实内容仅在受控调试场景短期保留 |

## 3. SQL 安全

### 3.1 防护链路

```text
用户问题
  -> DeepSeek function calling 生成结构化 SQL
  -> sqlglot AST 校验
  -> 表白名单检查
  -> 只允许单条 SELECT
  -> 自动注入 LIMIT
  -> MySQL 只读账号
  -> SET SESSION TRANSACTION READ ONLY
  -> MAX_EXECUTION_TIME 超时控制
```

### 3.2 禁止行为

| 行为 | 当前策略 |
| --- | --- |
| INSERT/UPDATE/DELETE | AST 校验拒绝 |
| DROP/ALTER/TRUNCATE | AST 校验拒绝 |
| 多语句 SQL | 拒绝 |
| 系统库访问 | 拒绝 |
| 非白名单表访问 | 拒绝 |
| 无限制大查询 | 自动 LIMIT |
| 慢查询 | MySQL hint 超时 |

说明：非白名单表访问已下沉到 `sqlglot` AST 校验层实现。校验器会收集 `WITH ... AS (...)` 中的 CTE 临时名称，避免把 CTE 名误判为真实表；除 CTE 名外，SQL 中出现的真实表名必须属于 `MYSQL_TABLES_WHITELIST`。

### 3.3 白名单

当前默认：

```text
affair, affair_task
```

后续如果开放更多表，必须同步：

1. `MYSQL_TABLES_WHITELIST`。
2. MySQL 业务只读账号的 `GRANT SELECT` 表范围。
3. `data/schema.md`。
4. NL2SQL few-shot。
5. SQL 评估集。
6. 权限与脱敏规则。

## 4. 数据安全

### 4.1 原始规范文档

`data/raw/` 中的真实规范文档不提交到 Git。

建议：

- 建立公开演示版脱敏文档。
- 明确文档来源、版本和可展示范围。
- README 中避免暴露真实敏感文档细节。

### 4.2 管理员观测接口

Phase 8.1 新增 `/api/admin/*` 只读观测接口，用于管理员前端查看会话、工具调用和审计摘要。当前边界：

- 仅允许 `role=admin` 的本地演示 token 访问，工程师用户返回 `403`。
- 只读取 MySQL 应用表，不修改业务数据。
- 只返回聚合数量和截断摘要，不返回完整真实问题、完整 SQL、完整检索片段或完整工具输出。
- 生产级 JWT、密码哈希、接口鉴权和管理员操作审计放在 Phase 9。

### 4.3 MySQL 业务数据

当前项目暂不考虑脱敏，允许展示人员姓名、部门名、设备名等字段。该约定仅适用于当前项目阶段和受控本地演示；后续如进入公开演示或生产环境，应重新评估脱敏和授权策略。

### 4.3 会话数据

当前会话数据保存在 MySQL 应用表：

```text
agent_conversation
agent_message
```

安全策略：

- 会话运行期数据不进入本地 SQLite，不随代码提交。
- 当前已由前端登录用户的 `user_id` 关联会话。
- `agent_conversation` 删除采用软删除，记录 `deleted_at` 与 `delete_status`。
- 管理端本地观测数据存储在浏览器 `localStorage`，仅用于开发演示。
- 后续生产化应按 admin/engineer 做服务端鉴权，并把删除原因写入审计表。

### 4.3.1 数据保留与删除策略

| 数据类型 | 默认保存时间 | 删除/归档策略 |
| --- | --- | --- |
| 普通会话正文 | 180 天 | 用户可主动删除个人会话；删除后前端立即不可见，30 天恢复期后物理删除 |
| 工具调用记录 | 1 年 | 用于问题定位和 Agent 行为追踪，到期归档或清理 |
| RAG 检索上下文 | 30-90 天 | 仅用于回答质量分析和检索效果优化 |
| 系统操作审计日志 | 2-3 年 | 包括登录、权限变更、文档管理、数据导出、模型配置修改和高风险工具调用 |
| 临时文件、缓存、未保存草稿 | 7-30 天 | 定时任务自动清理 |

所有删除操作必须记录操作者、删除时间、数据范围和删除原因。普通用户无权删除审计日志；管理员仅可按归档策略处理审计数据。

### 4.4 长期记忆

Phase 6.1 已实现长期记忆 MySQL 基础模型：

```text
user_memory
memory_event
```

缺失表建表 SQL 见 `docs/sql/phase6_1_memory_tables.sql`。

当前安全边界：

- 不自动沉淀对话内容，必须通过显式 API 写入。
- 每条记忆包含 `source`、`confidence`、`status`、`metadata` 和可选 `expires_at`。
- 记忆按 `user_id` 隔离，后续召回阶段必须保持用户隔离。
- 禁用和删除均记录 `memory_event`，删除采用软删除。
- 当前 API 仍是本地演示接口，生产化前需要接入服务端鉴权和管理员审计。

后续安全原则：

- 不默认沉淀敏感信息。
- 自动抽取前需要敏感信息过滤和可解释写入理由。
- 规范引用和记忆引用必须分开展示。
- 用户可在前端“记忆”工作区查看、禁用或删除自己的长期记忆。
- 管理员可按目标 `user_id` 查看和治理长期记忆；当前仍是本地演示鉴权，生产化前需要服务端校验角色和操作范围。
- Phase 8.2 新增记忆搜索、冲突提示和批量禁用。冲突提示只做规则提示，不自动合并；批量治理只支持禁用，不支持批量删除，且每条记忆都会写入 `memory_event`。

## 5. 前端安全

当前前端：

- 使用 Vite 本地代理到 `127.0.0.1:8000`。
- 使用 `fetch` 消费 POST SSE。
- Markdown 渲染关闭 HTML：`html: false`。
- 管理员日志暂存在浏览器 `localStorage`。

风险：

| 风险 | 当前处理 | 后续建议 |
| --- | --- | --- |
| XSS | MarkdownIt 禁用 HTML | 增加链接白名单和 DOMPurify |
| localStorage 泄露 | 仅本地演示 | 生产改后端持久化和鉴权 |
| CORS 过宽 | `allow_origins=["*"]` | 生产限制前端域名 |
| 本地 token 可伪造 | 当前仅演示 RBAC 入口 | 生产改签名 token 或服务端 session |

## 6. MCP 安全

MCP 工具是 Agent 可调用能力的扩展点，风险在于工具越权。

当前策略：

- `MCP_ENABLED` 默认关闭。
- 本地工具与 MCP 工具通过开关切换。
- MCP Server 只暴露受控领域工具。

后续生产化建议：

- MCP Server 增加鉴权。
- 记录每次 MCP 工具调用。
- 对远程 MCP 工具设置超时和返回大小限制。
- 禁止暴露文件系统写入类工具。

## 7. 权限模型规划

当前角色固定为两类：

| authority_flag | 角色 | 标识 |
| --- | --- | --- |
| `1` | 管理员 | `admin` |
| `0` | 工程师用户 | `engineer` |

### 7.1 管理员 admin 权限

| 权限域 | 权限 |
| --- | --- |
| 用户与角色 | 用户管理、角色分配 |
| 文档与知识库 | 文档上传、删除、更新；知识库重建；向量索引管理 |
| 系统配置 | 数据库连接配置、工具/API 配置、Agent 提示词配置、模型参数配置 |
| 观测与审计 | 查看所有问答记录、查看所有工具调用记录、查看系统运行日志、查看质量评估结果 |
| 数据治理 | 导出数据、删除异常数据、审核工程师提交的问题反馈 |

### 7.2 工程师用户 engineer 权限

| 权限域 | 权限 |
| --- | --- |
| 问答 | 提出业务问题；查询制度、规范、管理办法；查询设备相关知识 |
| 业务流程 | 查询巡检、维修、隐患、作业流程 |
| Agent 使用 | 查看知识库检索结果；使用 Agent 生成回答；使用 Agent 生成报告草稿 |
| 数据查询 | 查询允许范围内的数据库信息 |
| 历史与反馈 | 查看自己的历史问答记录；对回答进行反馈；标记回答是否正确；提交知识库补充建议 |

### 7.3 登录表

登录直接使用当前数据库中的 `user` 表：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `user_id` | int | 用户主键 |
| `username` | varchar(255) | 账号名 |

| `authority_flag` | int | `0=engineer`，`1=admin` |

## Phase 9.1 认证与权限生产化补充

本阶段新增签名 token 与密码哈希兼容策略：

- 登录成功后返回 HMAC-SHA256 签名的 JWT 风格 token，包含 `sub/user_id/username/role/authority_flag/iat/exp`。
- 新前端通过 `Authorization: Bearer <token>` 调用 `/api/auth/me` 与 `/api/admin/*`。
- 后端仍兼容 `?token=`，用于旧调用和测试过渡；后续可在生产环境逐步关闭。
- `user.password` 支持 `pbkdf2_sha256$iterations$salt$digest`，开发环境临时兼容明文密码。
- 生产环境必须设置强随机 `AUTH_SECRET_KEY`，并将 `AUTH_ALLOW_PLAINTEXT_PASSWORDS=false`。
- 管理员接口统一使用 `require_admin`，工程师用户访问返回 `403`。

密码迁移命令：

```powershell
uv run python scripts/hash_password.py --password <plain-password>
```

迁移 SQL 示例：

```sql
UPDATE `user`
SET password = 'pbkdf2_sha256$260000$...'
WHERE username = 'admin';
```

## Phase 9.2 数据保留与清理补充

数据清理任务已实现为显式脚本，默认 dry-run：

```powershell
uv run python scripts/cleanup_retention.py
uv run python scripts/cleanup_retention.py --execute --actor-id 1 --reason "monthly retention cleanup"
```

安全约束：

- 未加 `--execute` 时不修改数据库。
- 会话先软删除，恢复期后才物理删除。
- 长期记忆到期后只禁用，不物理删除。
- 工具日志和可选 RAG 检索上下文按保留期删除。
- 审计日志默认保留 3 年，正式执行后写入 `retention_cleanup` 审计记录。
- `action_detail` 记录清理原因、cutoff、影响数量和跳过项，便于追溯。

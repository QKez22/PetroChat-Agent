## 2. 规范文档数据

### 2.1 数据来源

当前项目使用石化规范文档构建知识库。原始文档放在 `data/raw/`，出于仓库体积和资料管理考虑不提交真实文件。规范文档名称以文件正式名称为准，不另设项目内简称。

### 2.2 解析策略

文档解析由 `src/petrochat/app/rag/parser.py` 完成，核心策略：

1. 优先读取 Word `outlineLvl`。
2. 若无结构化标题，则使用章节编号正则兜底。
3. 过长片段进行二次切分。
4. 每个 chunk 保留来源文档、章节号、章节路径、文档编码等元数据。

### 2.3 Chunk 模型

主要字段：

| 字段 | 含义 |
| --- | --- |
| `chunk_id` | 唯一 ID，由来源文档和章节派生 |
| `content` | 片段正文 |
| `source_doc` | 所属规范文档 |
| `section_number` | 章节号 |
| `section_path` | 章节路径 |
| `doc_code` | 文档编码 |
| `chunk_type` | clause/table/definition/other |

### 2.4 向量库

| 项 | 当前配置 |
| --- | --- |
| 向量库 | Chroma HTTP |
| Collection | `petrochat_specs` |
| Embedding | 阿里云百炼 `text-embedding-v3` |
| 维度 | 1024 |
| 入库脚本 | `scripts/ingest.py` |
| 幂等策略 | 按 `source_doc` 删除后重入 |

## 3. MySQL 业务数据

### 3.1 当前 NL2SQL 白名单

虽然 `data/schema.md` 中包含更多业务表，但当前代码默认白名单是：

```text
affair, affair_task
```

白名单配置项：

```text
MYSQL_TABLES_WHITELIST=affair,affair_task
```

该白名单同时用于两处：

1. Schema prompt 只暴露白名单表，降低模型生成越界 SQL 的概率。
2. SQL validator 基于 `sqlglot` AST 强制检查真实表名，拒绝访问非白名单表。

生产环境还需要在 MySQL `GRANT` 层只给业务只读账号授予白名单表的 `SELECT` 权限，避免配置或代码失误时扩大查询面。

### 3.2 表 `affair`

业务含义：事务定义表。每条记录表示一类周期性或规则性的业务事务。

重点字段：

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `affair_id` | 事务唯一 ID | 与任务表关联 |
| `specialty` | 专业 | 综合/动/静/电/仪/其它 |
| `area` | 区域 | 炼油/全厂/乙烯等 |
| `affair_name` | 事务名称 | 查询展示常用 |
| `trigger_status` | 调度状态 | 0 停止，1 运行，-1 删除 |
| `ticket_flag` | 是否需要开票 | 0 是，1 否 |
| `check_flag` | 是否需要线上检查表单 | 0 是，1 否 |
| `execution_department` | 执行部门 | 可能是逗号拼接 |
| `execution_role` | 执行角色 | 可能是逗号拼接 |
| `deadline` | 截止时间 | varchar，比较时需转换 |
| `next_trigger_time` | 下次触发时间 | varchar |
| `affair_level` | 事务级别 | 如总部级 |

### 3.3 表 `affair_task`

业务含义：任务执行表。每条记录表示事务下的具体执行任务，通常关联设备、提报人和执行信息。

重点字段：

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `task_id` | 任务 ID | 任务唯一标识 |
| `associated_affair_id` | 关联事务 ID | 对应 `affair.affair_id` |
| `task_name` | 任务名称 | 任务查询常用 |
| `task_properties` | 任务属性 | 任务分类 |
| `body_equipment_name` | 主体设备名称 | 设备维度分析 |
| `body_equipment_code` | 主体设备编码 | 设备维度分析 |
| `technical_id_code` | 主体设备位号 | 设备定位 |
| `operation_department` | 运行部 | 部门统计 |
| `execution_role` | 执行角色 | 执行人群 |
| `report_user` | 提报人 | 任务提报 |
| `report_time` | 提报时间 | datetime |
| `upload_auto` | 是否手动上传 | 中文枚举 是/否 |
| `maintenance_type` | 检修类别 | 检修分析 |

### 3.4 业务铁则

| 规则 | 说明 |
| --- | --- |
| 事务用 `affair` | 用户说“事务”时优先查 `affair` |
| 任务用 `affair_task` | 用户说“任务/执行/巡检/检修”时多查 `affair_task` |
| 关联键 | `affair.affair_id = affair_task.associated_affair_id` |
| 不用自增 ID 关联 | 两表 `id` 各自独立，无跨表意义 |
| `trigger_status` 只在事务表 | 任务表没有该字段 |
| `ticket_flag/check_flag` 反直觉 | 0 是，1 否 |
| 时间字段需区分 | varchar 时间需 `STR_TO_DATE`，datetime 可直接比较 |

## 4. 会话数据

当前会话数据使用 MySQL 应用表，与你截图中已有的表保持一致：

```text
agent_conversation
agent_message
agent_tool_log
agent_audit_log
SHORT_TERM_TURNS=6
```

表结构：

| 表 | 用途 |
| --- | --- |
| `agent_conversation` | 会话 ID、用户、标题、创建时间、更新时间、删除状态、保留策略 |
| `agent_message` | 用户/助手消息、正文、创建时间、删除时间 |

安全边界：

- 会话数据进入 MySQL 应用表，不再写入本地 SQLite。
- `agent_conversation` 删除采用软删除，设置 `deleted_at` 和 `delete_status`。
- 当前前端登录后会把数据库用户的 `user_id` 传给问答接口，用于会话隔离。
- 如果脚本直接调用接口并使用 `user_id=default`，后端会将其映射为开发用户 `0`。

### 4.1 会话与审计表结构

当前项目按以下应用表运行。你已创建 `agent_conversation`、`agent_message`、`agent_tool_log`、`agent_audit_log`；请确认 `agent_message` 至少包含下列字段：

```sql
CREATE TABLE agent_conversation (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(255),
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    expires_at DATETIME,
    deleted_at DATETIME,
    delete_status VARCHAR(50),
    retention_policy VARCHAR(50)
);

CREATE TABLE agent_message (
    id BIGINT PRIMARY KEY,
    conversation_id BIGINT NOT NULL,
    role VARCHAR(50),
    content TEXT,
    created_at DATETIME NOT NULL,
    deleted_at DATETIME
);

CREATE TABLE agent_tool_log (
    id BIGINT PRIMARY KEY,
    conversation_id BIGINT,
    user_id BIGINT,
    tool_name VARCHAR(100),
    input_summary TEXT,
    output_summary TEXT,
    status VARCHAR(50),
    error_message TEXT,
    created_at DATETIME NOT NULL,
    expires_at DATETIME
);

CREATE TABLE agent_audit_log (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    action_type VARCHAR(100),
    target_type VARCHAR(100),
    target_id VARCHAR(100),
    action_detail TEXT,
    ip_address VARCHAR(100),
    created_at DATETIME NOT NULL
);
```

### 4.2 数据保留策略

| 数据类型 | 默认保存时间 | 删除/归档策略 |
| --- | --- | --- |
| 普通会话正文 | 180 天 | 用户可主动删除个人会话；删除后前端立即不可见，30 天恢复期后物理删除 |
| 工具调用记录 | 1 年 | 用于问题定位和 Agent 行为追踪，到期归档或清理 |
| RAG 检索上下文 | 30-90 天 | 仅用于回答质量分析和检索效果优化 |
| 系统操作审计日志 | 2-3 年 | 包括登录、权限变更、文档管理、数据导出、模型配置修改和高风险工具调用 |
| 临时文件/缓存/未保存草稿 | 7-30 天 | 定时任务自动清理 |

所有删除操作必须记录操作者、删除时间、数据范围和删除原因，确保可追溯。

## 5. 长期记忆数据

Phase 6.1 新增长期记忆 MySQL 表，Phase 6.2/6.3 基础版已接入 Agent 召回和保守写入：

```text
user_memory
memory_event
```

当前实现数据模型、显式管理 API、调用前 active 记忆召回和问答结束后的规则写入。规则写入只处理用户显式表达的长期偏好/默认条件，不保存模型回答、SQL 原文或检索片段。
建表 SQL 见：

```text
docs/sql/phase6_1_memory_tables.sql
```

表结构：

| 表 | 用途 |
| --- | --- |
| `user_memory` | 用户长期记忆主体，保存偏好、常用条件、业务上下文等 |
| `memory_event` | 记忆生命周期审计，记录创建、更新、禁用、删除和访问事件 |

`user_memory` 重点字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 记忆 ID |
| `user_id` | 所属用户，用于隔离 admin / engineer 和不同工程师 |
| `memory_type` | 记忆类型，如 preference、business_context、query_filter |
| `content` | 记忆正文 |
| `source` | 来源，如 manual、agent_extracted、admin_import |
| `confidence` | 置信度，0-1 |
| `status` | active / disabled / deleted |
| `metadata_json` | 结构化扩展信息 |
| `expires_at` | 可选过期时间 |

`memory_event` 重点字段：

| 字段 | 含义 |
| --- | --- |
| `memory_id` | 对应记忆 ID |
| `event_type` | created / updated / disabled / deleted / accessed |
| `actor_id` | 操作者 ID |
| `reason` | 操作原因 |
| `payload_json` | 操作快照或补充信息 |

安全边界：

- 长期记忆进入 MySQL 应用表，不再写入本地 SQLite。
- 当前 API 使用显式 `user_id`，生产化前需要接入服务端鉴权。
- 删除采用软删除，将 `status` 置为 `deleted` 并记录 `memory_event`。
- 当前规则抽取只接受明确意图；后续 LLM 自动抽取前需要增加敏感信息过滤、用户确认、冲突合并和管理员审计策略。


## Phase 11 上下文与长期记忆数据更新

本阶段新增会话滚动摘要表，用于解决“只取最近 N 轮会漏掉早期关键信息”的问题。原始消息仍保存在 `agent_message`，prompt 只注入最近窗口和摘要，不会因为上下文裁剪删除历史记录。

新增数据对象：

| 数据对象 | 存储位置 | 作用 |
| --- | --- | --- |
| `agent_conversation_summary` | MySQL 应用库 | 保存当前会话的滚动摘要、`summarized_until_message_id` 摘要指针和版本 |
| `petrochat_memory_candidates` | Mem0 / Chroma collection | 候选记忆抽取隔离区，只承接 `infer=True` 自动抽取 |
| `petrochat_memories` | Mem0 / Chroma collection | 正式长期记忆语义索引，只同步 MySQL active 记忆 |

建表 SQL 位于：

```text
docs/sql/phase11_context_summary_tables.sql
```

如果已经按旧版 SQL 创建了 `covered_message_id` 字段，需要执行迁移脚本：

```text
docs/sql/phase11_context_summary_migration.sql
```

Docker demo 初始化脚本 `docker/mysql/init/001_demo_schema.sql` 已包含 `agent_conversation_summary`。如果是已有 MySQL 环境，需要手动执行上述 SQL 或迁移文件。

长期记忆仍以 `user_memory` 为事实主库，`memory_event` 为生命周期审计表。Mem0 只负责候选抽取和正式语义召回索引，召回后必须拿 `memory_id` 回 MySQL 校验 `user_id`、`status`、权限和类型。

完整流程见 [Phase11 上下文与长期记忆流程](Phase11-上下文与长期记忆流程.md)。

# 1.5 权限与安全文档

## 1. 安全目标

PetroChat-Agent 当前是学习和简历展示项目，但设计上需要体现生产化安全意识：

1. 不泄露 API Key。
2. 不允许 LLM 直接执行危险 SQL。
3. 不写业务数据库。
4. 不把图表 base64 塞进 LLM 上下文。
5. 不把用户会话和长期记忆无边界混用。
6. 前端清晰展示错误，不吞掉失败原因。

## 2. 密钥与配置安全

| 配置 | 用途 | 安全策略 |
| --- | --- | --- |
| `DEEPSEEK_API_KEY` | Chat / Reasoner 模型 | `.env`，不提交 |
| `DASHSCOPE_API_KEY` | Embedding | `.env`，不提交 |
| `MYSQL_PASSWORD` | 只读数据库连接 | `.env`，不提交 |
| `LANGSMITH_API_KEY` | Trace | `.env`，不提交 |

`.gitignore` 已忽略：

- `.env`
- `.env.local`
- `.env.*.local`
- `data/runtime/`
- `frontend/node_modules/`
- `frontend/dist/`

`/config` 只返回 key 是否配置，不返回明文。

## 3. SQL 安全

### 3.1 防护链路

```text
用户问题
  -> DeepSeek function calling 生成结构化 SQL
  -> sqlglot AST 校验
  -> 表白名单检查
  -> 只允许单条 SELECT
  -> 自动注入 LIMIT
  -> MySQL 只读账号
  -> SET SESSION TRANSACTION READ ONLY
  -> MAX_EXECUTION_TIME 超时控制
```

### 3.2 禁止行为

| 行为 | 当前策略 |
| --- | --- |
| INSERT/UPDATE/DELETE | AST 校验拒绝 |
| DROP/ALTER/TRUNCATE | AST 校验拒绝 |
| 多语句 SQL | 拒绝 |
| 系统库访问 | 拒绝 |
| 非白名单表访问 | 拒绝 |
| 无限制大查询 | 自动 LIMIT |
| 慢查询 | MySQL hint 超时 |

### 3.3 白名单

当前默认：

```text
affair, affair_task
```

后续如果开放更多表，必须同步：

1. `MYSQL_TABLES_WHITELIST`。
2. `data/schema.md`。
3. NL2SQL few-shot。
4. SQL 评估集。
5. 权限与脱敏规则。

## 4. 数据安全

### 4.1 原始规范文档

`data/raw/` 中的真实规范文档不提交到 Git。

建议：

- 建立公开演示版脱敏文档。
- 明确文档来源、版本和可展示范围。
- README 中避免暴露真实敏感文档细节。

### 4.2 MySQL 业务数据

当前项目会展示人员姓名、部门、设备等字段。生产化前需要确认：

- 是否为脱敏数据。
- 是否允许在前端展示人员姓名。
- 是否需要按部门限制查询。
- 是否需要审计每次查询。

### 4.3 会话数据

当前会话数据保存在本地 SQLite：

```text
data/runtime/petrochat_sessions.sqlite3
```

安全策略：

- 本地运行期数据不提交。
- 当前默认 `user_id=default`，仅适合单用户演示。
- 后续多用户必须引入登录和用户隔离。

### 4.4 长期记忆

长期记忆尚未实现，但规划安全原则：

- 不默认沉淀敏感信息。
- 每条记忆有来源、置信度、状态和删除入口。
- 规范引用和记忆引用必须分开展示。
- 记忆按 `user_id` 隔离。

## 5. 前端安全

当前前端：

- 使用 Vite 本地代理到 `127.0.0.1:8000`。
- 使用 `fetch` 消费 POST SSE。
- Markdown 渲染关闭 HTML：`html: false`。
- 管理员日志暂存在浏览器 `localStorage`。

风险：

| 风险 | 当前处理 | 后续建议 |
| --- | --- | --- |
| XSS | MarkdownIt 禁用 HTML | 增加链接白名单和 DOMPurify |
| localStorage 泄露 | 仅本地演示 | 生产改后端持久化和鉴权 |
| CORS 过宽 | `allow_origins=["*"]` | 生产限制前端域名 |
| 无登录 | 当前单用户演示 | 引入认证和 RBAC |

## 6. MCP 安全

MCP 工具是 Agent 可调用能力的扩展点，风险在于工具越权。

当前策略：

- `MCP_ENABLED` 默认关闭。
- 本地工具与 MCP 工具通过开关切换。
- MCP Server 只暴露受控领域工具。

后续生产化建议：

- MCP Server 增加鉴权。
- 记录每次 MCP 工具调用。
- 对远程 MCP 工具设置超时和返回大小限制。
- 禁止暴露文件系统写入类工具。

## 7. 权限模型规划

当前没有正式登录和权限系统。建议后续 RBAC：

| 角色 | 权限 |
| --- | --- |
| 普通用户 | 发起问答、查看自己的会话 |
| 管理员 | 查看会话观测、错误、工具调用和评估结果 |
| 数据管理员 | 管理 schema、评估集、数据源配置 |
| 系统管理员 | 管理密钥、MCP、部署和日志 |
